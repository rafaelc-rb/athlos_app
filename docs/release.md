## Versioning

Format: `MAJOR.MINOR.PATCH+build` (Semantic Versioning).

| Part | When to increment | Resets? |
| --- | --- | --- |
| **MAJOR** | Strategic milestone (first public release, backend migration, AI era) | MINOR and PATCH reset to 0 |
| **MINOR** | New user-facing feature or module | PATCH resets to 0 |
| **PATCH** | Bug fixes, performance, UI polish | ā€” |
| **build** (`+N`) | Every build uploaded to a store | **Never resets** ā€” always incrementing |

Version is set in `pubspec.yaml`:

```yaml
version: 0.1.0+1
#        ^^^^^ ^
#        ā”‚     ā””ā”€ā”€ build number (store identifier, always increasing)
#        ā””ā”€ā”€ semantic version (user-facing)
```

Both `versionName` (Android) and `CFBundleShortVersionString` (iOS) are derived from this automatically by Flutter.

See [Versioning Strategy](./context.md#versioning-strategy) for the full release roadmap.

## Android Signing

### Keystore

Release builds are signed with a dedicated keystore. The keystore file and credentials are **never committed to git** (protected by `android/.gitignore`).

| File | Location | Git |
| --- | --- | --- |
| Keystore | `android/app/athlos-release.jks` | Ignored (`**/*.jks`) |
| Credentials | `android/key.properties` | Ignored (`key.properties`) |

`key.properties` format:

```properties
storePassword=<password>
keyPassword=<password>
keyAlias=athlos
storeFile=app/athlos-release.jks
```

`build.gradle.kts` reads `key.properties` at build time. If the file is missing (e.g. CI without secrets, or a new dev machine), it falls back to debug signing so development builds still work.

### Keystore Safety

> **The keystore and its password must never be lost.** If you lose the keystore, you cannot update the app on the Play Store ā€” you would have to publish a new app with a new application ID. Back up the `.jks` file and the password in a secure location (password manager, encrypted drive, etc.).

### Generating a New Keystore (if needed)

```bash
keytool -genkey -v \
  -keystore android/app/athlos-release.jks \
  -keyalg RSA -keysize 2048 -validity 10000 \
  -alias athlos \
  -storepass <password> -keypass <password> \
  -dname "CN=Athlos, OU=Mobile, O=Athlos, L=Brazil, ST=BR, C=BR"
```

Then create `android/key.properties` with the matching credentials.

## Database Migrations (Drift)

### Schema Versioning

The database schema version is an integer in `app_database.dart`:

```dart
@override
int get schemaVersion => 1;  // increment when schema changes
```

- **Version 1** is the baseline for the first public release (1.0.0).
- Development used versions 1ā€“10 before release; the migration strategy detects these and recreates the database automatically.

### How Migrations Work

`onCreate` runs when a user installs the app for the first time ā€” it creates all tables and seeds catalog data.

`onUpgrade` runs when an existing user updates to a new version with a higher schema version. It uses `runMigrationSteps` for incremental, non-destructive migrations:

```dart
await m.runMigrationSteps(
  from: from,
  to: to,
  steps: {
    2: (m) async {
      // Example: add a new table for Diet module
      await m.createTable(foods);
    },
    3: (m) async {
      // Example: add a column to an existing table
      await m.addColumn(workouts, workouts.notes);
    },
  },
);
```

### Migration Rules

1. **Never modify an existing migration step** ā€” it may have already run on user devices
2. **Always increment `schemaVersion`** when adding a new step
3. **Test migrations** ā€” install the old version, add data, then update to the new version and verify data is preserved
4. **Seed data in migrations** ā€” if a new table needs catalog data (e.g. food catalog), call the seeder inside the migration step
5. **Use `transaction()` for complex migrations** ā€” Drift wraps each step in a transaction by default, but be explicit for multi-statement operations

### When to Add a Migration

| Change | Action |
| --- | --- |
| New table | `await m.createTable(newTable)` |
| New column (nullable or with default) | `await m.addColumn(table, table.newColumn)` |
| Remove column | `await m.alterTable(TableMigration(table))` |
| Rename column | `await m.alterTable(TableMigration(table, columnTransformer: {...}))` |
| New seed data for existing table | Insert rows inside the migration step |

## Building for Release

### Android (APK / App Bundle)

```bash

flutter build appbundle

# APK (for direct distribution)
flutter build apk
```

Output: `build/app/outputs/bundle/release/app-release.aab`

### iOS

```bash
flutter build ipa
```

Requires Xcode, an Apple Developer account, and provisioning profiles.
